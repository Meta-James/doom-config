* Process placement

- gptel and claude-code-ide.el both run as Emacs sub-processes/network clients on
  the *same host* as the buffer/project — correct for the local-machine workflow.
- For remote/TRAMP projects (SBCs, Armbian boards), the CLI-driven agent must run
  *on the remote host*, not locally against a TRAMP-mounted path. Untested — this
  is a Phase 6 acceptance criterion, not yet verified. See [[id:ai-evaluation][evaluation.org]].

* Context, provider, review, and credential flow

- *Context*: region/buffer/diagnostic context flows into gptel only via explicit
  user action; project-wide context flows into claude-code-ide.el via its own
  repo-walking, not via Emacs shoveling files at it.
- *Provider*: gptel's backend/model choice is explicit via =gptel-menu=; no silent
  fallback between providers.
- *Review*: agent proposes → Ediff/Magit surfaces the diff → per-hunk accept/reject
  → nothing is committed by the agent. Enforcement should be technical (Claude
  Code permission settings/hooks) where available, not merely instructional — see
  [[id:std-no-commit][standards.org]] and Phase 7 in [[id:docs-roadmap][roadmap.org]].
- *Credentials*: two-tier — the CLI agent uses its own subscription login;
  gptel/API-based tools use per-provider tokens from =pass= via Doom's =:tools
  pass=, never hardcoded. See [[id:ai-providers][providers.org]].
