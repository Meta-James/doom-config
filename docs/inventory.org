*vterm-module build, done once during Phase 6* (not shell-history-only — the
exact commands are captured in [[id:tshoot-vterm-module][troubleshooting.org]] and reproducible on any
machine with the toolchain above):
#+BEGIN_SRC sh
cd ~/.config/emacs/.local/straight/build-30.1/vterm
mkdir -p build && cd build
cmake -G 'Unix Makefiles' ..
make
#+END_SRC

*Doom's envvar file, regenerated once during Phase 7* (also not
shell-history-only — see [[id:tshoot-daemon-path-systemd][troubleshooting.org]]):
#+BEGIN_SRC sh
~/.config/emacs/bin/doom sync --env
#+END_SRC
Needed on any machine where Emacs (particularly a daemon) might be launched
by something other than an interactive login shell (systemd user session, a
display manager, etc.) — otherwise the daemon's =PATH= may not match the
user's real shell.

* Known environment issues
:PROPERTIES:
:ID:       inv-env-issues
:END:

- =$PATH= contains a literal, unexpanded =~/.config/emacs/bin= entry. Interactive
  bash resolves =doom= via its own tilde-aware command hashing, but any subprocess
  doing a raw =execvp= PATH search (e.g. =timeout doom ...=, and potentially some
  agent tool invocations) fails to find it. Not a Doom or Emacs issue — a shell
  profile issue. Affects anything that shells out to =doom= non-interactively,
  including future agent tool calls. Not yet fixed — out of scope for this
  project unless it starts blocking an agent workflow (see
  [[id:ai-troubleshooting][troubleshooting.org]]).
- The user's real Emacs daemon is launched by the systemd user session, not a
  login shell, so it inherited systemd's bare =PATH= (missing
  =~/.local/bin=, where =claude= lives) until Phase 7 fixed it by running
  =doom sync --env= — see [[id:tshoot-daemon-path-systemd][troubleshooting.org]] and the lifecycle table above.
  Fixed going forward for any new Emacs start; worth re-running =doom sync
  --env= (or =M-x doom/reload-env= on an already-running daemon) any time a
  new binary is added somewhere the daemon's environment wouldn't otherwise
  pick up.

* Reproduction checklist
:PROPERTIES:
:ID:       inv-reproduction-checklist
:END:

Not literally tested on a second machine (none available to this project) —
this is a documentation-completeness pass, not an executed test. Given a
fresh Ubuntu-like machine with Doom Emacs already bootstrapped against this
=$DOOMDIR=, reaching this config's actual working state requires, in order:

1. =doom sync= (picks up =packages.el=: gptel, copilot, claude-code-ide,
   perfect-margin).
2. Install prerequisites this project actually needed: =build-essential=
   +=cmake= (for vterm's native module — see the lifecycle table above),
   =npm=/=node= (for =copilot-language-server=, or accept the native-binary
   fallback), the =claude= CLI (official installer), =pass=+=gpg= with a
   trusted secret key already set up.
2. Compile vterm's native module (=cmake= + =make=, see the recipe above) —
   otherwise the first =(require 'vterm)= blocks on an interactive
   compile-confirmation prompt with nowhere to go in a headless context (see
   [[id:tshoot-vterm-module][troubleshooting.org]]).
3. =M-x copilot-install-server= (in-Emacs, not a shell step).
4. Create the =api/anthropic= and =api/openai= =pass= entries (manual,
   deliberately never scriptable — see [[id:ai-providers][providers.org]]).
5. Log in / authenticate the =claude= CLI itself (its own subscription login,
   outside this repo's credential system — see [[id:ai-providers][providers.org]]'s two-tier
   credential model).
6. Run =doom sync --env= once, so a daemon started by something other than
   an interactive shell (systemd, a display manager) still gets the right
   =PATH= — see the "Known environment issues" entry above.
7. =doom doctor=, confirm clean.

Known gaps in this checklist, honestly: it's reasoned from what this project
actually did on this one machine, not verified against a second one. Order
dependencies between steps 2-5 haven't been individually stress-tested (e.g.
whether =doom sync= must run before or after the =pass= entries exist).
